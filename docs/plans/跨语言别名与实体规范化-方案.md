# 功能方案:跨语言别名与实体规范化

> 状态:**规划中(未动代码)**
> 创建日期:2026-06-17
> 目标:女优/片商/标签常有中日英多种名称,**用户输入任一语言(尤其中文)都能找到对应数据**(很多源用日文),不至于因语言不匹配而搜不到;同一实体的多语言名在展示时合并为一个。

---

## 1. 现状

- ❌ 无任何别名/跨语言机制:`actors` 仅 `name`;`studio` 仅文本字段;`tags` 仅 name。
- 刮削是**番号驱动**(designation),不靠女优名;「按女优名搜索」是探索/演员模块的新能力,正需要跨语言映射。
- 同一女优(三上悠亜 / 三上悠亚 / Yua Mikami)、同一片商(エスワン / S1 / S1 NO.1 STYLE)在不同源是不同字符串,现在会被当成不同实体。

## 2. 两个层面(别混淆)

| 层面 | 解决什么 | 做法 |
|---|---|---|
| **规范化合并**(展示) | 同一实体多写法/多语言合并成一个 | canonical 实体 + 别名归并(参考 OpenAver `maker_mapping`、`canonicalize`) |
| **跨语言搜索展开**(查询) | 输入任一语言都能查到源数据 | 输入名 → 展开所有已知别名 → 用源偏好语言去查 |

用户强调的是**后者**:输入中文 → 找到日文内容。

## 3. 核心机制:同番号关联(最契合我们,零额外数据源)

我们本来就**多源刮削同一番号**——同一番号在不同语言的源会给出不同语言的名:

```
番号 SSIS-001
 ├─ JavBus(日)   → 女优「三上悠亜」  片商「エスワン ナンバーワンスタイル」
 ├─ 中文源        → 女优「三上悠亚」  片商「S1」
 └─ 英文源/罗马音 → 女优「Yua Mikami」
        ↓ 因同番号 → 判为同一实体的别名
   actor#X: { 三上悠亜(ja), 三上悠亚(zh), Yua Mikami(en) }
```

→ **跨语言别名随刮削自动积累**,不需要额外数据源。这是主力机制。

## 4. 辅助别名来源

| 实体 | 来源 |
|---|---|
| 片商 | OpenAver `maker_mapping.json`(405 条日英→规范名)作种子;同番号关联补中文 |
| 女优 | **Xslist(英/简中/繁中多语言名 + 别名/曾用名,女优库主力)**、**minnano-av(日文 + 旧艺名/曾用名)**、gfriends(头像为主,别名暂缺)、MetaTube `actor.aliases`(**内置 xslist provider,接 MetaTube 可间接获取**);同番号关联补足中文 |
| 标签 | OpenAver 式中日英同义词表(女僕=Maid=メイド)作种子 |
| 兜底 | AI 翻译(**仅兜底**,人名机翻易错,标记低置信,可人工校正) |

## 5. 数据模型

通用别名表(覆盖 actor/studio/tag):
```
entity_aliases
  entity_type  TEXT   actor / studio / tag
  entity_id    INTEGER
  name         TEXT
  lang         TEXT   ja / zh / en / unknown
  is_canonical BOOL   是否规范名(展示用)
  source       TEXT   关联来源(刮削源 / maker_mapping / 翻译)
  confidence   REAL   置信度(翻译来源偏低)
  UNIQUE(entity_type, name)
```
- 查询:`name → entity_id`(命中任一别名即定位实体)→ 取该实体所有别名。
- 展示:取 `is_canonical` 或按 UI 语言挑对应 `lang` 的名。

## 6. 搜索 / 刮削展开逻辑

```
用户输入名(任一语言)
  → 规范化(去空格/全半角/大小写)
  → entity_aliases 查命中 → entity_id
      ├─ 命中:取该实体所有别名 → 用数据源偏好语言(多为日文)去查/列作品
      └─ 未命中:按原名直接查;查到结果后把新名回写为别名(积累)
```

应用:女优名搜索(探索/演员模块)、片商维度合并、标签跨语言筛选。

## 7. 风险与注意

- **中文别名冷启动少**:依赖刮中文源积累;初期可用种子表 + 翻译兜底补。
- **人名机翻不可靠**:翻译仅兜底、低置信、可校正;优先同番号关联与社区表。
- **同名歧义**(不同人同名):结合番号上下文 / 源 id,合并要保守,宁可不合并。
- **误合并**:规范化匹配从严(精确/别名命中才合并),避免把不同实体并到一起。
- **Xslist / minnano-av 无公开 API**:需爬取(注意反爬/限速);或经 **MetaTube(内置 xslist provider)间接获取**,省去自维护别名库。

## 8. 分阶段

1. **数据模型**:`entity_aliases` 表 + 查询/回写接口。
2. **同番号关联**:刮削入库时,把各源给的女优/片商名按番号关联为别名(主力,自动积累)。
3. **种子表导入**:片商 `maker_mapping`、标签同义词、女优名库。
4. **搜索展开**:女优/片商/标签搜索接入别名展开(输入中文→查到日文数据)。
5. **展示规范化**:维度列表/详情按 canonical 合并,按 UI 语言显示。
6. **翻译兜底 + 人工校正**:可选。
