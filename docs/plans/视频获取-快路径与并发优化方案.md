# 视频链接 / 磁力获取 —— 快路径 + WebView 并发池 优化方案

> 状态:**规划中(未动代码)**
> 创建日期:2026-06-17
> 目标:把当前「单例 WebView、串行一个一个找」改造为「快路径 + 磁力 + WebView 池(默认 5 并发)」的并发架构。

---

## 1. 现状与问题

| 现状 | 位置 | 问题 |
|---|---|---|
| **单例 WebView** | `video_finder.rs` 固定 label `video_finder_webview`,打开新窗口前先 `close()` 旧的 + sleep 200ms | 一次只能找一个,串行 |
| **唯一获取方式 = WebView 拦截** | 注入 JS 拦截 `XHR/fetch/hls.js/媒体元素`,捕获 m3u8/mp4/ts | 慢(整页渲染 + 等播放器),即使是明文 m3u8 的站点也要走完整流程 |
| **事件全局** | `emit('video-finder-link', url)` | 多任务并发时无法区分链接属于哪个任务 |
| **无磁力获取** | — | 缺少磁力链接能力 |

**核心矛盾**:WebView 拦截鲁棒性强(对前端混淆免疫),但慢且重;而很多站点(如 JableTV)m3u8 是明文,完全不必动用 WebView。

---

## 2. 目标架构:三路获取 + WebView 池

对每个 `code + site` 任务,三种产出**并行、互不阻塞**地获取:

```
                       ┌─────────────────────────────────────────────┐
   任务队列(N 个视频) │                                             │
        │              │  ① 快路径(HTTP 正则)  ── 命中 m3u8 → 完成   │  HTTP 高并发(如 10)
        ├──────────────┤  ② 磁力获取(HTTP ajax)── JavBus 磁力       │  HTTP 高并发(如 10)
        │              │  ③ 快路径未命中 → 进 WebView 池             │
        │              └───────────────────────┬─────────────────────┘
        │                                       ▼
        │                         ┌──────── WebView 池(默认 5)────────┐
        │                         │ slot0 slot1 slot2 slot3 slot4       │
        └────────────────────────►│ 完成/超时/失败 → 关闭该 slot →      │
                                  │ 队列下一个任务顶上                  │
                                  └─────────────────────────────────────┘
```

- **快路径**和**磁力**都是纯 HTTP,轻量、可高并发,不占 WebView。
- **WebView 池**是稀缺资源(5 个),只处理快路径啃不动的(CDN 重组/解包都失败、需用户交互或 JS 动态加载、强反爬的站)。

---

## 3. 模块设计

### 3.1 快路径(HTTP 直抓 m3u8)
- 每个站点实现一个可选的 `try_fast_resolve(code) -> Option<Vec<VideoLink>>`,复用现有 `fetcher`(wreq + TLS 指纹 + curl 风格反爬)。
- **明文站(JableTV)**:GET 详情页 → 正则 `https://.+\.m3u8` 直接拿。秒级。
- **CDN 重组站(MissAV,推荐解法,参考 competitor-analysis/nassav.md)**:**不必完整解 JS packer**——
  1. GET 详情页(URL 依次试 `/cn/{code}-chinese-subtitle` → `-uncensored-leak` → `/cn/{code}` → `/dm13/cn/{code}`,顺带得到字幕/无码标记);
  2. 正则从 packed 代码抠 UUID 碎片:`m3u8\|([a-f0-9\|]+)\|com\|surrit\|https\|video`,按 `|` 分隔**反转重组**为 UUID;
  3. 拼固定 CDN:`https://surrit.com/{uuid}/playlist.m3u8`;
  4. 请求该 master m3u8,解析 `BANDWIDTH`/`RESOLUTION`,按画质设置选最高/最低清晰度子流。
  → 全程纯 HTTP,免解包、免 WebView,秒级。
- **解包站(兜底)**:CDN 重组失效时,才退回 JS packer 解包(`eval(p,a,c,k,e,d)` 还原)→ 抽 `source='...m3u8'`;仍失败则进 WebView 池。
- 站点能力声明:`fast_path: 明文正则 / CDN 重组 / 解包 / 不支持`。不支持的站直接进 WebView 池,不浪费一次 HTTP。

### 3.2 磁力链接获取 + 排序(独立维度,详见 competitor-analysis/codeseek.md、hyperq-jav.md)
- JavBus:详情页提取 `gid`/`uc`/`img` 三个 JS 变量 → 请求 `uncledatoolsbyajax.php`(带 Referer,`floor` 用随机数)→ 用 `scraper` 选择器 `a[href^="magnet"]` 解析磁力表格。
- 纯 HTTP,与视频链接获取**完全并行**,复用现有 javbus 源 + fetcher + scraper。
- 产出 `MagnetItem[]` 入库,详情页展示,可一键推送到现有 download 模块(N_m3u8DL-RE)。

**磁力字段**(每条磁力解析出):
```
MagnetItem {
  link: String,       // magnet:?xt=...
  size: String,       // 文件大小(原始文本,如 "5.2GB")
  size_bytes: u64,    // 解析为字节,用于排序
  date: String,       // 发布日期
  is_hd: bool,        // 高清(名字含 hd/1080/720 或行内高清标记)
  has_subtitle: bool, // 字幕(行内含「字幕」标记 / btn-warning)
}
```

**排序设计(参考 hyperq/jav:字幕 > 高清 > 文件大小)**:
- 默认排序键(降序优先级):① `has_subtitle` ② `is_hd` ③ `size_bytes`。
- 即:有字幕的排最前,其次高清,再按体积从大到小;据此自动**推荐/默认选中最优磁力**。
- UI 支持手动切换排序维度(字幕优先 / 清晰度优先 / 体积优先 / 日期),并以徽标展示「字幕」「HD」标签。
- 批量场景下「一键取最优」即取排序后的首条。

### 3.3 WebView 池 + 调度器(本方案重点)
- **多实例**:label 由单例 `video_finder_webview` 改为 `video_finder_webview_{slot}`(slot 0..N-1)。
- **池管理**:`Semaphore(并发数)` 或空闲 slot 队列;一个任务占一个 slot。
- **顶替**:slot 上的任务在 **拿到链接 / 超时 / 页面被关 / 失败** 任一情况 → 关闭该 WebView → 释放 slot → 调度器立刻从待办队列取下一个任务、在该 slot 启动新 WebView。
- **超时**:每个 WebView 任务独立超时(可配,如 20s);超时即判失败并顶替。
- **并发数可配**:默认 5,设置项可调(低配机器调小)。

## 4. 关键技术问题(必须解决)

| 问题 | 说明 | 方案 |
|---|---|---|
| **事件归属** | 现在 `video-finder-link` 全局,5 个 WebView 并发时无法区分链接属于哪个任务 | 注入 JS 时带 `task_id`/`slot`,emit `{task_id, url}`;后端按 task_id 路由到对应任务的结果收集器 |
| **可见性 / CF** | 当前 WebView 可见(为过 Cloudflare)。5 个同时弹窗体验极差 | **默认隐藏/离屏静默加载**;仅当某 slot 检测到 CF 挑战(已有 `video-finder-cf-state` 事件)时,**只把那一个**显示出来让用户过,过完转回后台 |
| **资源占用** | 每个 WebView2 实例占数十~上百 MB 内存 | 并发数可配;空闲即关闭回收;不常驻 |
| **slot 复用 vs 重建** | 复用同一窗口换 URL 可能残留状态/拦截器 | 倾向「用完关闭、新任务新建窗口」,干净隔离(与现有单例关闭逻辑一致) |
| **结果聚合** | 一个任务可能同时来快路径 m3u8 + WebView 链接 + 磁力 | 按 task_id 聚合;m3u8 命中后可提前取消该任务的 WebView slot,让它顶给别人 |
| **取消/批量停止** | 批量任务中途取消 | 调度器持有所有 slot 句柄,统一 close |

## 6. 分阶段实施

1. **阶段 1 — 事件带 task_id**:先把 WebView 注入 JS 和事件改成携带 task_id(为并发铺路,单例下也兼容)。
2. **阶段 2 — WebView 池 + 调度器**:多 slot + 信号量 + 超时顶替 + 默认隐藏/遇 CF 显示。
3. **阶段 3 — 快路径**:JableTV 等明文站 HTTP 直抓;命中跳过 WebView。
4. **阶段 4 — 磁力**:JavBus ajax 获取 + 入库 + UI + 推送下载。
5. **阶段 5 — 设置与调优**:并发数/超时可配;MissAV 解包快路径(可选)。

## 7. 配置项(建议默认值)

- `webview_pool_size`: 5
- `webview_timeout_secs`: 20
- `fast_path_enabled`: true
- `magnet_enabled`: true
- `magnet_sort`: "subtitle_hd_size"(默认:字幕 > 高清 > 体积;可选 size / date / hd)
- `http_concurrency`(快路径/磁力): 10

## 8. 风险

- WebView 池内存占用随并发数线性上升,需给默认值与上限。
- 「默认隐藏 + 遇 CF 才显示」是体验关键也是难点;若实现不了离屏,退而求其次为「最小化/单独区域」。
- 快路径与 WebView 对同一任务可能产出不同质量的流,需定义优先级(快路径明文流通常即可用)。
- 站点改版导致快路径失效时,需能自动回退 WebView(保留鲁棒性)。
- 磁力、快路径强依赖站点结构,易随改版失效,需日志可观测。
