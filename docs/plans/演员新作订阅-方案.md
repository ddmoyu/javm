# 功能方案:演员新作订阅

> 状态:**规划中(未动代码)**
> 创建日期:2026-06-17
> 优先级:🟢 低(JavHub #3,hyperq 旁证)
> 目标:订阅喜欢的女优,后台定时检查新作,有新片提示。**高复用、易实现**。

---

## 1. 原理(一句话)

订阅女优 → 后台定时(应用启动时 + 每日)抓该女优**作品全集**(演员模块已有的 JavBus star 页抓取)→ 与**上次快照对比** → 多出来的番号 = 新作 → 红点 / 桌面通知。

**不需要新爬虫**,就是「定时跑已有的作品全集抓取 + 比对差异」。

## 2. 为什么易实现:几乎全是复用

| 依赖 | 来自 |
|---|---|
| 订阅女优表 `favorite_actors` | `探索模块-全网发现方案.md` 已规划 |
| 女优作品全集抓取 | `演员模块-功能方案.md` 已规划 |
| 新增的只有 | `last_checked_at` + 作品番号快照 + diff + 提示 |

## 3. 设计

### 3.1 数据
- `favorite_actors` 加 `last_checked_at`、`known_codes`(上次抓到的番号集合,或单独 `actor_work_snapshot` 表)。
- 新作记录:`new_works`(actor_id + code + 发现时间 + 已读),供「新作」列表与红点。

### 3.2 定时检查
- 触发:应用启动 + 后台定时(`tokio` interval,周期可配,如每 12/24h);避免频繁(尊重源站)。
- 流程:对每个订阅女优 → 抓作品全集 → `当前番号集 - known_codes` = 新增 → 写 `new_works` + 更新 `known_codes` + `last_checked_at`。
- 限流:复用 `反爬工具箱-方案.md`(限速)+ 并发控制,分散请求。

### 3.3 提示
- 桌面系统通知(Tauri notification)+ 应用内红点/「新作」聚合页。
- 新作项可:加入想看清单 / 直接刮削 / 下载(复用探索闭环)。

## 4. 复用 / 改动

| 层 | 改动 |
|---|---|
| `db/` | `favorite_actors.last_checked_at` + `known_codes`;`new_works` 表 |
| 后端 | 定时任务 + diff 逻辑;`get_new_works()` |
| 前端 | 「新作」聚合页 + 红点;订阅管理(已在演员/探索) |
| 复用 | 作品全集抓取、收藏表、反爬限速、桌面通知 |

## 5. 分阶段

1. **快照 + 手动检查**:订阅女优存 known_codes,手动「检查新作」按钮跑 diff。
2. **后台定时**:启动 + interval 自动检查。
3. **通知 + 新作页**:桌面通知 + 聚合页 + 一键加想看/下载。

## 6. 风险

- 定时抓取频繁会增加被封风险 → 周期保守 + 限速 + 分散;仅订阅的女优,量可控。
- 女优作品全集源覆盖/准确性 → 同演员模块。
